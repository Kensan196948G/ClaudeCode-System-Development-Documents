# 05 マイクロサービス設計（Microservice Design）

---

## 概要

モノリシックアーキテクチャからマイクロサービスへの設計・移行を  
Claude Code が支援するためのプロンプトです。

---

## Claude Code 起動コマンド

```
以下のサービスをマイクロサービスアーキテクチャで設計してください。

【対象システム】
[既存システムの概要]

【分割方針】
- ドメイン駆動設計（DDD）の境界コンテキストに基づく分割
- チーム組織に合わせた分割（Conway's Law）
- データの独立性確保

【技術要件】
- サービス間通信: [REST / gRPC / イベント駆動]
- サービスディスカバリ: [Kubernetes / Consul]
- API ゲートウェイ: [Nginx / Kong / AWS API Gateway]
- 分散トレーシング: [Jaeger / Zipkin]

各サービスのAPI仕様とデータモデルをOpenAPI形式で出力してください。
```

---

## サービス分割の判断基準

### 分割すべき場合

```
✅ 独立したデプロイサイクルが必要
✅ 異なるスケーリング要件がある
✅ チームが独立して開発・運用できる
✅ 技術スタックを分けたい
✅ 障害を局所化したい
```

### 分割すべきでない場合

```
❌ チームが小さい（2-pizza rule 未満）
❌ データが密結合で分離困難
❌ レイテンシ要件が厳しい（サービス間通信のオーバーヘッド）
❌ 運用コストを増やしたくない
```

---

## サービス間通信パターン

### 同期通信（REST / gRPC）

```
適用場面:
- リクエスト/レスポンスが即時必要
- 単純なCRUD操作
- 強整合性が必要

注意点:
- 依存サービスの障害が伝播する
- Circuit Breaker パターンで対策
```

### 非同期通信（イベント駆動）

```
適用場面:
- 結果を待たなくてよい処理
- 複数サービスへの通知
- 最終的整合性で許容できる

メッセージブローカー:
- Apache Kafka（高スループット）
- RabbitMQ（柔軟なルーティング）
- AWS SQS/SNS（マネージド）
```

## 分散トレーシング設定

```yaml
# OpenTelemetry 設定例
tracing:
  exporter: jaeger
  endpoint: http://jaeger:14268/api/traces
  sampling_rate: 0.1  # 本番: 10%サンプリング
  service_name: order-service

# 必須ヘッダー伝播
propagation:
  - tracecontext
  - baggage
```

## チェックリスト

### 設計フェーズ
- [ ] サービス境界が明確に定義されている
- [ ] データの所有権が各サービスに割り当てられている
- [ ] サービス間の通信方式が決定されている
- [ ] 障害シナリオが検討されている

### 実装フェーズ
- [ ] Circuit Breaker が実装されている
- [ ] ヘルスチェックエンドポイントがある
- [ ] 分散トレーシングが設定されている
- [ ] 各サービスが独立してデプロイ可能である
