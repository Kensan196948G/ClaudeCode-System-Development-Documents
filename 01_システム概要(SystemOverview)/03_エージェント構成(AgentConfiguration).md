# 03 エージェント構成（Agent Configuration）

---

## 概要

ClaudeCode 自律開発システムは **7体の専門エージェント** がオーケストレーションにより連携し、  
ソフトウェア開発のライフサイクル全体をカバーします。

---

## エージェント一覧

### Agent 1: 要件定義エージェント（Requirements Agent）

**役割**: ビジネス要求・要件定義

```
起動キーワード: 「requirements-agentとして」
```

| 責務 | 詳細 |
|------|------|
| 要求分析 | Issue・仕様書からタスクを整理 |
| ユースケース定義 | 機能一覧・スコープ・前提条件の抽出 |
| 受け入れ基準 | 完了条件・例外ケース・非機能要件の明文化 |
| 出力物 | タスクブリーフ、要件定義書 |

---

### Agent 2: アーキテクトエージェント（Architect Agent）

**役割**: システム設計・モジュール分割・I/F設計

```
起動キーワード: 「architect-agentとして」
```

| 責務 | 詳細 |
|------|------|
| アーキテクチャ設計 | レイヤ構造・依存関係・責務分担 |
| I/F設計 | API仕様・ERD・データモデル |
| リスク評価 | 変更影響範囲・互換性・デプロイ戦略 |
| 出力物 | 設計サマリ、アーキテクチャ図 |

---

### Agent 3: 実装エージェント（Implementation Agent）

**役割**: コード実装・リファクタリング専任

```
起動キーワード: 「implementation-agentとして」
```

| 責務 | 詳細 |
|------|------|
| コード生成 | 既存方針に沿った新機能実装 |
| 一括修正 | 関連ファイルの一括更新 |
| PR分割 | 小さなPR単位への分割提案 |
| 出力物 | 実装コード、プルリクエスト |

---

### Agent 4: テスト・品質保証エージェント（Testing Agent）

**役割**: テスト戦略設計・テストコード作成・品質ゲート

```
起動キーワード: 「testing-agentとして」
```

| 責務 | 詳細 |
|------|------|
| テスト設計 | 単体・結合・E2Eテスト観点の洗い出し |
| ギャップ分析 | 既存テストの網羅漏れ検出 |
| テストコード作成 | 新規・変更分のテスト実装 |
| 出力物 | テストコード、カバレッジレポート |

---

### Agent 5: セキュリティ／コンプライアンスエージェント（Security Agent）

**役割**: セキュリティレビュー・規約適合チェック

```
起動キーワード: 「security-agentとして」
```

| 責務 | 詳細 |
|------|------|
| 脆弱性検出 | インジェクション・認証・ログ漏洩パターン検出 |
| OSS監査 | ライセンス・依存ライブラリリスク確認 |
| ベストプラクティス | CIS/NIST基準とのギャップ指摘 |
| 出力物 | セキュリティレポート、修正案 |

---

### Agent 6: インフラ／DevOpsエージェント（DevOps Agent）

**役割**: CI/CD・IaC・運用自動化専任

```
起動キーワード: 「devops-agentとして」
```

| 責務 | 詳細 |
|------|------|
| パイプライン設計 | GitHub Actions / Argo CD 等の設計・更新 |
| IaC管理 | Terraform・Ansible・Helm レビュー・生成 |
| 障害分析 | パイプライン失敗ログからの原因特定 |
| 出力物 | CI/CDパイプライン、IaCコード |

---

### Agent 7: ドキュメント・ナレッジ管理エージェント（Docs Agent）

**役割**: ドキュメント生成・ナレッジ管理

```
起動キーワード: 「docs-agentとして」
```

| 責務 | 詳細 |
|------|------|
| ドキュメント生成 | PR差分・設計メモからMarkdown化 |
| チェンジログ作成 | リリースノート・Runbookの下書き生成 |
| テンプレート管理 | 各エージェント成果物テンプレートの標準化 |
| 出力物 | 技術文書、運用手順書 |

---

## エージェント起動パターン

### 標準開発パイプライン（新機能開発）

```
1 → 2 → 3 → 4 → 5 → 6 → 7
要件 設計 実装 テスト セキュリティ DevOps ドキュメント
```

### バグ修正

```
1 → 3 → 4 → 5 → 7
（設計・DevOps省略可）
```

### セキュリティ対応

```
5 → 3 → 4 → 7
```

### インフラ変更

```
2 → 6 → 5 → 7
```

## 並列実行（Fleet モード）とサブエージェント

独立したタスクは並列実行で効率化できます。

```bash
# 例: 設計と実装を並列起動
「architect-agentとimplementation-agentを並列起動して設計と実装を同時進行」
```

### サブエージェントによる自動並列化

Claude Code 2.0 では、メインエージェントが自動的にサブエージェントに仕事を委譲することができます：

```
メインエージェント（オーケストレーター）
  ├── 実装サブエージェント  → src/ のコード実装
  ├── テストサブエージェント → tests/ のテスト作成
  └── ドキュメントサブエージェント → docs/ の更新
```

詳細は [サブエージェント設計](../02_起動・設定(StartupConfig)/09_サブエージェント設計(SubagentDesign).md) を参照してください。

## Hooks によるエージェント制御

各エージェントの動作にHooksを設定して、より細かい制御が可能です：

```json
// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": { "tool_name": "bash" },
        "hooks": [{ "type": "command", "command": "echo 'bash実行前チェック'" }]
      }
    ]
  }
}
```

詳細は [Hooks設定ガイド](../02_起動・設定(StartupConfig)/06_Hooks設定ガイド(HooksConfig).md) を参照してください。
