# 02 Plan Mode（計画モード）ガイド

> **概要**: Claude Code の Plan Mode（計画モード）は、実装前に詳細な計画を立て、ユーザーが承認してから実行に移る構造化ワークフローです。

---

## Plan Mode とは

Plan Mode は `EnterPlanMode` / `ExitPlanMode` ツールによって制御される、**計画→承認→実行** の2ステップワークフローです。

```
┌─────────────────────────────────────────────────────────────┐
│                    Plan Mode ワークフロー                     │
│                                                             │
│  1. EnterPlanMode  →  計画立案フェーズ                       │
│       ↓ (ファイル読み込み・調査のみ、コード変更なし)           │
│  2. 計画を文書化  →  ユーザーに提示                          │
│       ↓ (ユーザーが内容を確認)                               │
│  3. ExitPlanMode  →  ユーザーが承認                          │
│       ↓                                                     │
│  4. 実装フェーズ  →  計画に従ってコード変更                   │
└─────────────────────────────────────────────────────────────┘
```

---

## Plan Mode の特徴

| 特徴 | 詳細 |
|------|------|
| **安全性** | Plan フェーズではコード変更・ファイル作成を行わない |
| **透明性** | 実装前に全ステップをユーザーが確認できる |
| **承認制** | ExitPlanMode でユーザーが明示的に承認して初めて実行 |
| **修正可能** | 承認前であれば計画を自由に修正・方針変更できる |

## Claude Code での使い方

### 方法1: `/plan` コマンド

```bash
# Plan Mode を開始
/plan

# その後に具体的なタスクを入力
認証システムをJWTからOAuth2に移行してください
```

### 方法2: タスク記述の先頭に "plan:" を付ける

```
plan: ユーザー管理APIのエンドポイントを追加する
```

### 方法3: プロンプトで計画を要求

```
実装前に詳細な計画を立ててください。変更するファイルと手順を教えてください。
```

## Plan Mode 活用パターン

### パターン1: 大規模リファクタリング

```markdown
## Plan Modeで依頼する例

「src/auth/ ディレクトリ全体をリファクタリングしたい。
まず影響範囲と変更計画を教えてください。実装前に確認します。」

## AIが提示する計画例

### 変更ファイル一覧
| ファイル | 変更内容 | 影響度 |
|---------|---------|-------|
| src/auth/jwt.ts | JWT → OAuth2 移行 | 高 |
| src/auth/middleware.ts | 認証ミドルウェア更新 | 高 |
| tests/auth/*.test.ts | テストケース追加 | 中 |

### 実行順序
1. 新しい OAuth2 クライアントクラス作成
2. 既存JWTコードをラッパーで互換維持
3. ミドルウェア段階的移行
4. テスト追加・既存テスト修正
5. JWTコード削除（フラグ確認後）
```

### パターン2: 新機能追加

```markdown
「ユーザープロフィール編集機能を追加したい。
バックエンドAPIとフロントエンドUIの両方の実装計画を先に教えてください。」
```

### パターン3: データベースマイグレーション

```markdown
「users テーブルに email_verified カラムを追加したい。
既存データへの影響とロールバック手順を含む計画を立ててください。」
```

## Plan Mode での AskUserQuestion 活用

Plan フェーズでは、実装方針について選択肢を提示してユーザーに選ばせます。

```markdown
## 方針の選択肢

**Q: 認証方式をどれにしますか？**
- A) JWT（シンプル・現行互換）
- B) OAuth2（標準・拡張性高）
- C) Session-based（セキュリティ最強）

選択後に詳細計画を確定します。
```

## Plan Mode 内での禁止事項

Plan フェーズでは以下を行いません：

| 禁止操作 | 理由 |
|---------|------|
| ファイルの Edit/Write | 承認前の変更防止 |
| git commit/push | 未確認コードの送信防止 |
| パッケージのインストール | 意図しない依存追加防止 |
| データベース変更 | 本番データへの影響防止 |

Plan フェーズで許可される操作：
- ファイルの Read（読み込み・調査）
- Glob/Grep（検索・分析）
- Bash（読み取り専用コマンド: ls, cat, git status など）

## CLAUDE.md での Plan Mode 設定

```markdown
# 大規模変更時のルール

## Plan Mode 必須条件
以下の場合は必ず Plan Mode で計画を先に提示すること：
- 変更ファイルが3つ以上
- データベーススキーマの変更
- 認証・認可ロジックの変更
- 本番環境への影響がある変更

## 計画書フォーマット
| 項目 | 内容 |
|------|------|
| 変更ファイル | ファイルパス・変更内容 |
| 影響範囲 | 依存する機能・テスト |
| 実行順序 | ステップバイステップ |
| ロールバック手順 | 問題発生時の復旧方法 |
| リスク評価 | 高/中/低 + 対策 |
```

## settings.json での Plan Mode 設定

```json
{
  "planMode": {
    "autoEnterForLargeChanges": true,
    "fileThreshold": 3,
    "requireApprovalFor": [
      "database",
      "auth",
      "production"
    ]
  }
}
```

## Triple Loop との統合

```
Monitor Loop → 優先課題特定
    ↓
[Plan Mode 開始]
    ↓
Architect が計画立案（Opus 4.7 推奨）
    ↓
ユーザーが計画承認
    ↓
[Plan Mode 終了 → Build Loop 開始]
    ↓
Developer が実装（Sonnet 4.6）
    ↓
Verify Loop → 品質確認
```
